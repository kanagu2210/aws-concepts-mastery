# Concept Mastery Pipeline — Claude Code Briefing

## What This Project Is
A **generic** Python pipeline that generates a concept-based learning curriculum.
The subject (AWS, Kubernetes, Piano, anything) is defined entirely by `input/syllabus.yaml`.
Each concept flows through 6 stages: outline → curate → expand → narrate → render → publish.
Output per concept: a teaching script (.md), an SVG whiteboard (.html), and a styled reading page (.html/.pdf).

## The Syllabus (`input/syllabus.yaml`)

This is the **single file** that makes the pipeline subject-specific.
To create a different course, copy this file and change every value.

### Schema terminology

The schema uses universal terms that work for any mastery journey:

| Term | Instead of | Why |
|------|-----------|-----|
| `phases` | layers | Every mastery journey progresses through phases |
| `areas` | domains | Topic areas — neutral and universal |
| `milestones` | exams | Proficiency checkpoints (certs, grades, skill levels) |
| `applied` | interviews | Real-world practical relevance |
| `concept_type` | spine_type | Pedagogical shape of each concept |
| `core_idea` | concept_spine | The one-sentence essence |
| `entity_field` | service_field | The domain-specific "thing" (service/skill/technique) |

### Hierarchy: milestones → phases → areas

```
MILESTONES  define WHAT you're working toward
    └── each milestone spans certain PHASES (via `phases` field)
PHASES      define the ORDER you learn things
    └── each phase covers certain AREAS (via outline_plan)
AREAS       define the GROUPING of related concepts
    └── each concept sits in one phase + one area + serves N milestones
```

### Sections

| Section | What it defines |
|---------|----------------|
| `project_name`, `series_name`, `subject` | Identity |
| `expert_role` | Role description injected into all prompts |
| `milestones` | Proficiency targets with names, depth, phase ranges, badge colors |
| `phases` | Progression stages with names, descriptions, targets, allowed types, colors |
| `areas` | Topic area categories |
| `concept_schema` | Allowed concept_types, tiers, roles, entity field name |
| `analogy_sources` | Scenarios for teaching analogies |
| `teaching_notes` | Style rules injected into narration prompts |
| `coverage_map` | Checklist of what must be covered |
| `outline_plan` | Generation plan — each entry = one API call (phase × area intersection) |

## Concept Schema (spine fields)

Each concept in `concept_spines.yaml` has these fields:

| Field | Purpose |
|-------|---------|
| `id`, `slug`, `title` | Identity |
| `phase` | Which learning phase (1, 2, 3, ...) |
| `phase_name` | Name of the phase ("Foundations", etc.) |
| `area` | Topic area ("Security and Compliance", etc.) |
| `concept_type` | Pedagogical shape: mental_model, concept, contrast, pattern, bridge |
| `milestones` | Which milestones this concept serves: [CCP, SAA, SAP] |
| `applied` | Whether this has real-world practical use (true/false) |
| `concept_tier` | foundation, core, extension |
| `cognitive_role` | definition, contrast, model, application |
| `core_idea` | One-sentence essence of the concept |
| `<entity_field>` | Domain-specific entity (e.g. `aws_service: "IAM"`) |
| `notes` | Coverage references |

## How Prompts Work

All prompts in `01_prompts/` are **generic templates** with `{{PLACEHOLDER}}` variables.
When loaded via `_load_prompt()`, syllabus values are injected:

| Placeholder | Source |
|-------------|--------|
| `{{PROJECT_NAME}}` | `syllabus.project_name` |
| `{{SUBJECT}}` | `syllabus.subject` |
| `{{EXPERT_ROLE}}` | `syllabus.expert_role` |
| `{{ANALOGY_SOURCES}}` | Formatted from `syllabus.analogy_sources` |
| `{{MILESTONE_DEPTH_MAP}}` | Formatted from `syllabus.milestones` |
| `{{PHASE_DESCRIPTIONS}}` | Formatted from `syllabus.phases` |
| `{{TEACHING_NOTES}}` | `syllabus.teaching_notes` |
| `{{COVERAGE_MAP}}` | `syllabus.coverage_map` |
| `{{ENTITY_FIELD}}` | `syllabus.concept_schema.entity_field` |
| `{{MILESTONE_TAG_CSS}}` | Generated from milestone badge colors |
| `{{SERIES_NAME}}` | `syllabus.series_name` |

**Never hardcode subject-specific content in prompts.** Use placeholders.

## How to Run

```bash
# Full pipeline for one concept
python 02_pipeline/run.py --spine 78

# Resume from a specific stage
python 02_pipeline/run.py --spine 78 --start-stage curate

# Range of concepts
python 02_pipeline/run.py --from 71 --to 81

# All concepts in a phase
python 02_pipeline/run.py --layer 2

# All concepts, skip existing outputs
python 02_pipeline/run.py --all --resume

# Whiteboard (runs separately)
python 02_pipeline/run_whiteboard.py --spine 78

# Publish to PDF
python 02_pipeline/run_publish.py --spine 78

# Dashboard
python 02_pipeline/build_dashboard.py --open
```

**Stage flags:** `--start-stage expand | narrate | curate | render`
**All scripts support:** `--spine N`, `--spines N N N`, `--from N --to N`, `--layer N`, `--all`, `--resume`, `--overwrite`, `--dry-run`

## Key File Responsibilities

### `input/syllabus.yaml` — THE subject-specific input
Everything subject-specific lives here. All scripts and prompts read from it.

### `utils/config.py` — config + syllabus loader
- Loads `config.yaml` (model, paths, PDF settings)
- Loads `input/syllabus.yaml` and exposes as `syllabus` dict
- Provides helpers: `syllabus_phase_names()`, `syllabus_milestone_tags()`, `syllabus_outline_plan()`, etc.

### `02_pipeline/pipeline_common.py` — THE central hub
- Path helpers: `concept_path()`, `narrated_path()`, etc.
- `_part_subdir(spine_id)` — derives `phase_N/part_N` from syllabus `outline_plan`
- Concept loaders and CLI selection logic

### `utils/api.py`
- `call_claude(user_content, model=None, max_tokens=None)` → `(text, in_tok, out_tok)`
- Auto-switches to streaming when `max_tokens > 8000`
- Retry: up to 5 attempts with exponential backoff

## Conventions — Follow These When Making Changes

### Subject-specific content
- **All subject-specific content lives in `input/syllabus.yaml`**
- **All prompts use `{{PLACEHOLDER}}` variables** — never hardcode subject content
- Scripts derive enums, phase names, milestone tags from the syllabus at runtime

### Paths
- **Never hardcode** phase/part folder names
- Always use path helpers from `pipeline_common.py`
- Part folders are derived from the syllabus `outline_plan`

### API calls
- All Anthropic API calls must go through `utils/api.py`

### Cost logging
- Every API call result must be logged via `utils/usage_log.py`
- Never skip logging even in error paths

### CLI flags
- All selection logic is handled by `_select_spines()` in `pipeline_common.py`
- Never reimplement selection logic in individual scripts

---

## Do Not
- **Do not hardcode subject-specific values in scripts** — use syllabus
- **Do not hardcode subject-specific content in prompts** — use `{{PLACEHOLDER}}` variables
- **Do not edit `input/concept_spines.yaml` directly** — it's generated
- **Do not hardcode paths** — always use pipeline_common.py helpers
- **Do not add API calls outside `utils/api.py`**
- **Do not skip usage logging**
- **Do not add `[Board:]` markers to narration**

---

## Narration Style Rules (for editing narrated .md files)
- Analogies use ONLY the sources defined in `syllabus.yaml`
- Opens with a session frame, closes with a 2–4 sentence summary, then a landing line
- `[Pause: ...]` markers are intentional — never remove them
- No banned phrases: "Let's explore", "In summary", "Key insight"
- Depth stages control length: D1 = 600–900w, D3 = ~1400w, D5 = 2000–2800w
